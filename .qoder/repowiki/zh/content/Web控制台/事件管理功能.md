## 核心组件
- 事件定义与查询
  - 后端通过事件定义处理器提供事件列表查询接口，返回事件名、显示名、描述、状态及首次/最近出现时间等元数据。
  - 前端事件管理页基于该接口渲染表格，并支持项目筛选。
- **更新** 事件模式管理
  - 新增事件模式管理功能，支持配置事件的必带参数（schema_required_props）、事件状态（status）和模式锁定（schema_locked）。
  - 通过调试器页面提供可视化的事件 Schema 配置界面，支持实时保存和锁定规则。
- 事件 Schema 与类型
  - 统一事件结构由服务端模型与 JSON Schema 定义，包含事件类型、事件名、标识符、时间戳、SDK 信息与自定义属性。
  - 三端 SDK（Web/Android/iOS）上报结构与 Schema 对齐。
- 分析与监控
  - 后端分析接口提供趋势、Top 事件、漏斗与留存等能力，前端控制台事件趋势页调用趋势接口进行可视化。
- 数据流
  - SDK 将事件写入本地存储或内存缓冲，周期性或达到阈值后发送至采集器；采集器写入消息队列，消费者从队列读取并落库/写入分析引擎。

## 事件模式管理功能

### 功能概述
事件模式管理是 AeroLog 的核心治理功能，提供以下能力：
- **必带参数配置**：为每个事件定义必需的参数列表，确保上报数据完整性
- **事件状态管理**：控制事件是否参与分析和展示，支持启用/禁用切换
- **模式锁定机制**：锁定事件规则，防止误修改影响数据质量
- **实时验证**：在事件采集阶段进行 Schema 验证，及时发现问题

### 核心概念
- **schema_required_props**：JSONB 数组格式，存储事件的必带参数列表
- **schema_locked**：布尔值，控制事件模式是否可以被修改
- **status**：事件状态，0 表示禁用，非 0 表示启用
- **requiredProps**：后端解析后的字符串数组，供验证逻辑使用

### 配置流程
1. 在调试器页面选择目标事件
2. 在"必带参数"文本框中输入必需参数，每行一个参数 key
3. 点击"保存事件规则"按钮保存配置
4. 如需锁定规则，点击锁定按钮启用模式锁定

```mermaid

未锁定 --> 已锁定 : 锁定规则
已锁定 --> 未锁定 : 解锁规则

启用 --> 禁用 : 设置状态=0
禁用 --> 启用 : 设置状态≠0
启用 --> 已锁定 : 保存并锁定
```

## 事件 Schema 验证与调试

### 验证机制
事件 Schema 验证在消费者端进行，主要包含以下检查：

1. **事件状态检查**：如果事件状态为禁用（status=0），标记为错误级别
2. **必带参数检查**：验证事件是否包含所有必需参数
3. **参数类型检查**：验证参数类型是否符合定义
4. **枚举值检查**：验证参数值是否在允许的枚举范围内

### 调试功能
调试器页面提供完整的 Schema 验证和调试功能：

- **事件流**：显示最近接收的事件，包括事件名、结果、用户标识和参数摘要
- **Schema 问题**：列出所有 Schema 验证问题，包括事件、参数、期望类型、实际类型和严重程度
- **统计信息**：显示最近事件数、Schema 告警数、问题记录数和锁定参数数

### 问题分类
- **错误级别**：事件被禁用、必带参数缺失等严重问题
- **警告级别**：参数类型不符、枚举值不在允许范围内等潜在问题

## 性能考虑
- 上报与缓冲
  - SDK 支持批量大小与刷新间隔配置，减少网络开销；Android SDK 具备本地持久化与周期刷新，提升稳定性。
- 查询优化
  - 分析接口支持时间范围与聚合粒度参数，建议按需缩小时间窗口与合理设置粒度。
- 存储与索引
  - 事件定义表按项目+事件名建立唯一索引，保障唯一性与查询效率。
  - **新增** 新增调试事件表和 Schema 问题表的索引，优化查询性能。
- 并发与吞吐
  - 采集器与消费者分别负责高并发写入与消费，建议根据流量调整批次大小与并发度。
- **新增** Schema 验证性能
  - 验证逻辑在消费者端执行，建议合理设置验证规则数量，避免过高的验证开销。

## 故障排查指南
- 事件无法出现在事件管理页
  - 检查项目筛选是否正确，确认已创建项目且事件定义已入库。
  - 核对事件名是否符合命名规范（长度与字符），避免因校验失败导致入库异常。
- 趋势/Top 事件为空
  - 确认时间范围参数是否合理，检查事件是否在目标时间范围内产生。
  - 若使用控制台事件趋势页，确认已选择项目与事件。
- Android 自动埋点无效
  - 确认初始化配置已开启自动埋点选项，检查应用生命周期与活动回调注册逻辑。
- 上报失败或延迟
  - 检查 SDK 的服务器地址与 Token 配置，确认网络连通性与服务端可达性。
  - 关注 SDK 的调试日志输出，定位失败原因。
- **新增** 事件 Schema 验证问题
  - 检查调试器中的 Schema 问题列表，确认事件状态、必带参数和参数类型是否正确。
  - 如果事件被标记为禁用，检查事件状态配置。
  - 如果出现必带参数缺失问题，确认 SDK 上报时是否包含了所有必需参数。
- **新增** 模式锁定问题
  - 如果无法修改事件规则，检查是否已锁定模式。
  - 解锁后重新配置事件 Schema 规则。

### 事件模板与标准化管理建议
- 模板化
  - 建议为常见业务场景（如支付、注册、浏览）提供事件模板，统一事件名、显示名与关键属性。
- 标准化
  - 强制事件命名规范与属性命名规范，避免歧义；定期清理废弃事件，保持事件集合整洁。
- 版本演进
  - 对于需要演进的事件，建议新增事件并保留旧事件一段时间，确保历史数据可追溯。
- **新增** 模式管理
  - 建立事件模式审查流程，确保必带参数配置的合理性。
  - 制定模式锁定决策标准，平衡数据质量控制与开发灵活性。

### 事件模式管理最佳实践
- **必带参数配置**
  - 仅配置真正重要的业务参数作为必带参数，避免过度严格影响数据收集。
  - 参数命名应简洁明确，避免使用缩写或容易混淆的名称。
  - 定期审查必带参数列表，移除不再使用的参数。
- **事件状态管理**
  - 新建事件默认设置为启用状态，确保不影响正常业务。
  - 重大业务变更前先禁用相关事件，待验证后再重新启用。
  - 建立事件停用的审批流程，避免误操作影响生产环境。
- **模式锁定机制**
  - 在生产环境稳定运行一段时间后考虑锁定模式。
  - 锁定前必须完成充分的测试和验证，确保不会影响数据质量。
  - 建立模式解锁的紧急预案，快速响应突发问题。
- **监控与告警**
  - 建立 Schema 验证问题的监控告警，及时发现数据质量问题。
  - 定期分析调试器中的问题报告，持续优化事件模式配置。
  - 监控事件状态变化，确保业务逻辑与配置保持一致。
